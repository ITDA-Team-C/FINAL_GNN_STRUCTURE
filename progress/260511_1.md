## 1. 데이터 파이프라인 (Phase 1)

### 1.1 입력
- **원본 CSV**: `data/raw/yelp_zip.csv` (608,458 reviews, 410MB)
- 컬럼: user_id, prod_id, rating, label, date, text, tag

### 1.2 단계별 흐름
```
load_yelpzip       → data/interim/raw_data.csv
label_convert      → data/interim/labeled_data.csv  (-1→1, 1→0)
sampling           → data/processed/sampled_reviews.csv (50k nodes + split)
feature_engineering→ data/processed/{features.npy, node_samples.csv, feature_meta.json}
build_relations    → data/processed/edge_index_dict.pt (6 relations)
relation_quality   → outputs/metrics/relation_quality.json
```

### 1.3 검수 결과 (Step A)

| 파일 | 결과 | 비고 |
|---|---|---|
| sampling.py | ✅ | hybrid dense 방식, sample→split 순서 OK |
| feature_engineering.py | ⚠️ | 기존: 전체 데이터로 fit. **재작성 → train-only fit** |
| evaluate.py + threshold.py | ✅ | threshold valid에서 결정, test 1회 평가 |

**수정 사항**: `feature_engineering.py`를 train-only fit으로 재작성. `feature_meta.json["fit_scope"] == "train_only"` 박제.

### 1.4 검증된 수치

- **원본 fraud_ratio**: 13.22% (608,458 중 80,439)
- **샘플**: 50,000 nodes (target=25k였으나 union 486k라 max_nodes=50,000 상한 적용)
- **샘플 fraud_ratio**: 11.16% (Train 3,572 / Valid 893 / Test 1,116)
- **Split**: 32k / 8k / 10k = 64/16/20 (stratified, seed=42)
- **Feature shape**: (50000, 140) = 128 SVD + 12 numeric
- **Numeric 12D**: rating_norm, review_length(_log), user_*(3), product_*(3), days_since_first_review, month_sin/cos

### 1.5 6 Relations + Fraud Edge Lift (train labels only)

| Relation | edges | avg_degree | isolated | FF | FN | **lift** |
|---|---|---|---|---|---|---|
| R-U-R | 49,754 | 5.73 | 0.653 | 0.0157 | 0.0040 | **1.26** |
| R-T-R | 87,228 | 5.58 | 0.375 | 0.0211 | 0.2040 | **1.69** |
| R-S-R | 597,432 | 25.71 | 0.070 | 0.0139 | 0.1780 | **1.12** |
| **R-Burst-R** | 33,672 | 3.65 | 0.631 | 0.0244 | 0.1892 | **1.96 ★** |
| R-SemSim-R | 330,132 | 13.56 | 0.026 | 0.0139 | 0.1735 | **1.12** |
| R-Behavior-R | 550,136 | 23.43 | 0.061 | 0.0091 | 0.1468 | **0.73 ⚠️** |

해석:
- **R-Burst-R가 가장 강한 fraud 신호** (random 대비 96% 증가)
- **R-Behavior-R는 noisy** (random보다도 약함, 0.73)

### 1.6 발견한 버그 (포팅 코드)

**`build_behavior.py`**: 기존 코드가 review×review (50000×50000) cosine을 시도 → 18.6 GiB 메모리 요구로 OOM.
- 수정: user 단위 cosine top-k (chunked) → user pair → review pair 확장
- `max_reviews_per_user=3`으로 edge 폭발 방지

## 2. 모델 구조 (Phase 2)

### 2.1 핵심 모듈 (신규 작성 4개 + 포팅)

| 파일 | 종류 | 책임 |
|---|---|---|
| `src/models/skip_cheb_branch.py` | 신규 | ChebConv K=3 + residual skip |
| `src/models/gated_relation_fusion.py` | 신규 | 노드별 softmax α + entropy regularizer |
| `src/filtering/care_neighbor_filter.py` | 신규 | 노드 feature cosine top-k 필터링 (label-free) |
| `src/models/cage_carerf_gnn.py` | 신규 | 위 3개 통합 + main/aux classifier |
| `src/models/cage_rf_gnn_cheb.py` | 포팅 | 기존 통합 구현 (v8 Skip + v9 Two-Stage 호환) |
| `src/models/baseline_{mlp,gcn,gat,graphsage}.py` | 포팅+수정 | union of 6 relations 사용으로 패치 |
| `src/models/losses.py` | 포팅 | FocalLoss + AuxiliaryLoss + WeightedBCELoss |

### 2.2 CAGE-CareRF 통합 모델 (cage_carerf_gnn)

```
Input: x (N, 140), edge_index_dict (6 relations)
        │
[offline] ▼ CARE neighbor filter (feature cosine top-k)
6 filtered relations
        │
        ▼ SkipChebBranch ×6 (per relation, ChebConv K=3 + residual)
(N, 6, 128)
        │
        ▼ GatedRelationFusion (softmax α per node)  OR  Mean Fusion
(N, 128)
        │
        ▼ Projection (Linear→ReLU→Dropout)
        ▼ Main Classifier → main_logit
        + Auxiliary heads → aux_logits per relation

Loss = Focal(main, y) + 0.3 × mean_r BCE(aux_r, y)
threshold @ valid PR-curve → Test 1회 평가
```

설정 토글: `use_skip` / `use_gating` / `use_aux_loss` / `active_relations` / `care_inline`.

## 4. 옵션 A 채택 — Lean 모델로 FINAL 결정

### 4.1 결정 근거

ablation 결과 **Gating + Custom Relations 모두 음(-)의 효과**. 그러나 단순 결합은 측정 안 됨 → Lean = "둘 다 끔" 학습 필요.

### 4.2 ⚠️ 규정 위반 발견 및 수정

- **초기 Lean**: basic 3개만 (R-U-R, R-T-R, R-S-R)
- **문제**: 대회 규정 "기본 ≥ 1 AND **커스텀 ≥ 1**" 위반
- **수정**: R-Burst-R(lift 1.96, 가장 강한 custom 신호) 필수 포함

### 4.3 Lean 변종 3개 (FINAL 후보)

| 변종 | active_relations | n_rel | n_custom | 근거 |
|---|---|---|---|---|
| **Lean-4** | rur, rtr, rsr, burst | 4 | 1 | rule-compliant minimal (lift 1.96만) |
| **Lean-5** | rur, rtr, rsr, burst, semsim | 5 | 2 | drop noisy Behavior (lift 0.73) |
| **Lean-6** | 6 all | 6 | 3 | = ablation_no_gating (이미 측정됨) |

모두 `gating=off + Skip=on + Aux=on + CARE=on`. 학습 후 best 1개를 FINAL로 확정.

### 4.4 예상 결과

- Lean-4: PR-AUC ~0.30~0.32 추정
- Lean-5: 비슷하거나 살짝 위/아래 (SemSim lift 1.12)
- Lean-6: ablation_no_gating과 동등하면 ~0.3117

## 8. 메모 / 의사결정 기록

### 8.1 Sample size 50,000 유지 결정
- target=25,000이었으나 union이 486k라 max_nodes=50,000 상한으로 reduce됨
- 규정 범위(10k~50k) 내 — 유지 결정 (재샘플링 시간 절약)
- 보고서에 "후보군 union 내부 보정용 reduce" 명시 필요

### 8.2 PC-GNN sampler 메인 제외 결정
- 이유: 대회 규정의 subgraph sampling 절차와 혼동 가능성
- 클래스 불균형은 Focal + class_weight + threshold tuning으로 대체
- `configs/cage_carerf*.yaml`에 `sampler.enabled: false` + note 박제

### 8.3 옵션 A (Lean) vs 옵션 C (v1 유지) 선택
- ablation에서 Gating·Custom 음(-)의 효과 입증 → 옵션 A 채택
- 단 단순 Lean (basic 3개)은 규정 위반 → R-Burst-R 필수
- Lean-4/5/6 비교 학습으로 최적 위치 탐색

### 8.4 추가 미세 튜닝 안 함 결정
- Hard Mining, CARE top-k 차등, Behavior 개선, Gating reg 등 검토
- 사용자 결정: "0.001 올리려고 추가 안 함" → Lean 비교만으로 종료
- 시간 절약 + 보고서 narrative 단순화
